# Spring Boot sample app (intentionally vulnerable, multi-module)

A small, **real** multi-module (reactor) Spring Boot Maven app used as the end-to-end test
bed for `dep-remediation`. Its poms are committed in a deliberately **vulnerable "before"
state** so the sample is reusable — runs of the tool should be made against a copy.

> ⚠️ This app pins **old, vulnerable** dependency versions on purpose. Do not use it as a
> starting template for real work.

## Layout

```
spring-boot-sample/        aggregator (packaging pom) — managed/transitive security pins go here
├── sample-core/           library module: the direct + property findings
└── sample-web/            Spring Boot app module: the managed + transitive findings
```

`sample-web` depends on `sample-core`, so several findings resolve in **both** modules — the
reactor-wide resolution check (`verify` at the root) covers every module.

## Seeded findings — one per resolution class

| Library | Where / how | Resolution class | Before → recommended |
|---------|-------------|------------------|----------------------|
| `org.apache.commons:commons-text` | `sample-core`, literal `<version>` | **direct** | 1.9 → 1.10.0 |
| `org.apache.commons:commons-collections4` | `sample-core`, `${commons-collections4.version}` | **property** | 4.1 → 4.4 |
| `org.yaml:snakeyaml` | `sample-web`, versionless (Boot parent BOM) | **managed** | 2.2 → 2.3 |
| `org.apache.tomcat.embed:tomcat-embed-core` | `sample-web`, via `spring-boot-starter-web` | **transitive** | 10.1.24 → 10.1.25 |

The matching advisory is `advisory.xlsx`, regenerated by `make_advisory.py` (it also seeds
noise rows — a different owner, a container/OS row, and a base-image row — to exercise the
parser's filter chain).

## Fix targeting (the reactor wrinkle)

A `<dependencyManagement>` pin only affects its own module subtree and **cannot override a
module's explicit `<version>`**, so each finding is routed to the right pom:

- **direct / property** (`commons-text`, `commons-collections4`) → edit **`sample-core/pom.xml`**.
- **managed / transitive** (`snakeyaml`, `tomcat-embed-core`) → pin in the **aggregator
  `pom.xml`** so every module inherits it.

Today the routing is manual, using `--skip` to send the right subset to each pom (see below).
Having the engine choose the correct parent/module pom automatically is **auto fix-targeting**
(plan §13.4) — the open work this sample motivates.

## Try it end to end

```bash
# from the repo root
python examples/spring-boot-sample/make_advisory.py   # (re)generate advisory.xlsx + advisory-breaking.xlsx

# work on a COPY so the committed sample stays vulnerable
cp -r examples/spring-boot-sample /tmp/sample && cd /tmp/sample
ADV=advisory.xlsx

# route direct + property to sample-core
dep-remediation fix sample-core/pom.xml --from-advisory $ADV --app sample-app \
    --skip org.yaml:snakeyaml --skip org.apache.tomcat.embed:tomcat-embed-core --apply
# route managed + transitive pins to the aggregator
dep-remediation fix pom.xml --from-advisory $ADV --app sample-app \
    --skip org.apache.commons:commons-text --skip org.apache.commons:commons-collections4 --apply
# verify the whole reactor at the aggregator root
dep-remediation verify . --from-advisory $ADV --app sample-app
```

A captured happy-path run lives in [`SHAKEOUT.md`](SHAKEOUT.md). `make_advisory.py` also
emits `advisory-breaking.xlsx` (one finding recommends a non-existent version) to drive the
**build-failure recovery loop** — a captured recovery run is in [`RECOVERY.md`](RECOVERY.md).

## Requirements

- JDK 17+ and Maven (a project `mvnw`/`mvnw.cmd` wrapper is included at the root; system
  `mvn` also works). The committed "before" state builds green (vulnerable, but compiles):

```bash
cd examples/spring-boot-sample && ./mvnw clean install   # or: mvn clean install
```
